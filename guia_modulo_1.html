Setup — Parte 1

Ambiente de desenvolvimento

Antes de qualquer coisa, precisamos deixar o sistema operacional e o editor prontos. Esses dois passos são feitos uma única vez — depois disso, é só abrir o VS Code e programar.

O que vamos configurar aqui Linux (via WSL2 ou nativo) e VS Code com as extensões certas. Git, Ruby e Claude Code têm configurações próprias nos módulos correspondentes.

Passo 1 — Linux

Ruby e Rails vivem no mundo Linux. Servidores de produção rodam Linux, ferramentas de desenvolvimento foram criadas para Linux, e a experiência de desenvolvimento é muito mais fluida nesse ambiente. Por isso, antes de qualquer coisa, precisamos de um terminal Linux na sua máquina.

Você tem duas opções — escolha a que faz mais sentido para o seu momento:

Recomendado para Windows
WSL2 — Windows Subsystem for Linux

O WSL2 instala um Linux completo dentro do seu Windows, sem precisar formatar o computador ou criar dual boot. Você continua usando o Windows normalmente e acessa o Linux sempre que precisar programar.

Como instalar o WSL2

Abra o PowerShell como Administrador (botão direito no menu Iniciar → Windows PowerShell (Admin)) e execute:

PowerShell (Admin) 
wsl --install

Esse único comando instala o WSL2 com Ubuntu. Quando terminar, reinicie o computador. Na primeira abertura do Ubuntu, ele vai pedir para você criar um nome de usuário e senha — use algo simples que você vai lembrar.

Dica Após reiniciar, procure por "Ubuntu" no menu Iniciar. Esse é o terminal Linux que você vai usar para tudo. Fixe-o na barra de tarefas.

Verificando a instalação

Terminal Ubuntu 
$ lsb_release -a
Ubuntu 24.04 LTS
Linux nativo
Ubuntu instalado diretamente

Se você já usa Linux nativamente ou quer instalar o Ubuntu como sistema principal, está no caminho ideal. O ambiente de desenvolvimento é idêntico ao de produção — sem camadas intermediárias.

Recomendamos o Ubuntu 24.04 LTS — versão estável com suporte longo e ótima compatibilidade com as ferramentas que usaremos.

Download

Baixe a imagem ISO em ubuntu.com/download/desktop e siga o instalador.

Dual boot Se quiser manter o Windows e o Linux no mesmo computador, é possível — basta optar pelo dual boot durante a instalação.

Aula 1

O que é Git e por que usar

Git é um sistema de controle de versão. Pense nele como um histórico de salvamentos do seu código — você pode ver o que mudou, quando mudou, quem mudou e, se precisar, voltar para qualquer ponto anterior. Sem Git, se você apagar algo por acidente ou introduzir um bug, pode não ter como recuperar.

Foi criado em 2005 por Linus Torvalds — o mesmo criador do Linux — para gerenciar contribuições de centenas de desenvolvedores no kernel do Linux. Se funciona nessa escala, funciona para tudo.

Git vs GitHub São coisas diferentes. Git é o sistema de controle de versão que roda na sua máquina. GitHub é o serviço online onde você armazena e compartilha repositórios Git. É como a diferença entre o Word (programa) e o Google Drive (onde você salva os arquivos).

Como o Git enxerga seu projeto

O Git não salva cópias de arquivos — ele salva snapshots (fotografias) do estado do projeto em cada commit. Cada snapshot sabe exatamente o que mudou em relação ao anterior, quem fez a mudança e quando.

Cada bolinha é um commit — uma fotografia do projeto naquele momento. O HEAD sempre aponta para o commit mais recente onde você está trabalhando. Se precisar voltar para o C3, você pode — e o código vai estar exatamente como estava naquele momento.

Os três estágios do Git

Essa é a parte que a maioria das pessoas pula e depois fica confusa. O Git não salva automaticamente — você controla exatamente o que entra em cada salvamento. Para isso, existe um fluxo de três etapas:

Por que existe o Stage? Porque nem sempre você quer salvar tudo ao mesmo tempo. Imagine que você corrigiu um bug e começou uma feature nova no mesmo arquivo. Com o stage, você pode commitar só a correção do bug agora e deixar a feature para depois — mantendo o histórico organizado e com sentido.

Vendo os três estágios na prática

Terminal — git status é seu melhor amigo 
# Depois de criar ou modificar um arquivo
$ git status
On branch main
Changes not staged for commit:
  modified:   calculadora.rb     ← no working directory

# Adicionar ao stage
$ git add calculadora.rb
$ git status
Changes to be committed:
  modified:   calculadora.rb     ← no stage, pronto para commitar

# Criar o commit
$ git commit -m "feat: adiciona calculadora"
$ git status
nothing to commit, working tree clean  ← tudo salvo!

Aula 4

Branches — linhas paralelas

Um branch é uma linha independente de desenvolvimento. Imagine que a main é a versão "oficial" do projeto. Você cria um branch para trabalhar em uma funcionalidade nova — sem mexer no que já está funcionando. Quando terminar e testar, você junta tudo de volta.

Trabalhando com branches

Terminal — branches 
# Ver todos os branches
$ git branch
* main

# Criar e entrar no branch ao mesmo tempo
$ git checkout -b quest02
Switched to a new branch 'quest02'

# Confirmar em qual branch você está
$ git branch
  main
* quest02    ← asterisco = branch atual

# Voltar para a main
$ git checkout main

# Deletar um branch (depois do merge)
$ git branch -d quest02

# Comparar diferenças entre dois branches
$ git diff main quest02

Merge — juntando tudo

Quando seu trabalho no branch está pronto, você junta de volta na main. Nas Quests isso vai acontecer via Pull Request no GitHub — você abre o PR, o mentor revisa o código e faz o merge. Mas é bom entender como funciona localmente também:

Terminal — merge 
# Estando na main, juntar o branch quest02
$ git checkout main
$ git merge quest02
Updating 9d8e7f6..a3f2c1b
Fast-forward
 quests/quest02.txt | 2 ++
 1 file changed, 15 insertions(+)
E se houver conflito? Conflito acontece quando dois branches modificaram a mesma linha do mesmo arquivo. O Git pausa o merge e marca os conflitos no arquivo. Você abre o arquivo no VS Code, escolhe qual versão manter, salva e faz um novo commit. O VS Code mostra os conflitos de forma visual — muito mais fácil do que resolver no terminal.

Aula 5

Repositório remoto e Pull Request

O repositório remoto é a versão do projeto que fica no GitHub. Você trabalha localmente e sincroniza com o remoto através de push e pull.

Terminal — sincronização 
# Enviar commits para o GitHub
$ git push origin main

# Enviar um branch específico
$ git push origin quest03

# Baixar e aplicar mudanças do GitHub
$ git pull origin main

# Ver os remotos configurados
$ git remote -v
origin  https://github.com/gitDbits/mentoria_barbara.git (fetch)
origin  https://github.com/gitDbits/mentoria_barbara.git (push)

Pull Request — o fluxo de revisão

Um Pull Request (PR) é um pedido para juntar seu branch na main. Antes do merge, o mentor lê seu código, deixa comentários e você pode corrigir antes de aprovar. É assim que times profissionais trabalham.

Dica de ouro Sempre faça git pull origin main antes de começar a trabalhar e antes de fazer git push. Isso garante que você está sincronizado e evita conflitos desnecessários.
Quest 03 ⭐ Bônus · Pull Request completo
O fluxo real de um time
+120 XP
  1. Atualize seu repositório local: git pull origin main
  2. Crie o branch: git checkout -b quest03
  3. Crie o arquivo quests/quest03.txt com o texto: "Iniciando a Quest 03." Faça o primeiro commit: git commit -m "feat: cria quest03"

Aula 6

Desfazendo erros

Uma das maiores vantagens do Git é poder desfazer. Errou? Calma. Tem solução para quase tudo.

Terminal — situações comuns 
# Errei a mensagem do último commit (antes do push)
$ git commit --amend -m "mensagem corrigida"

# Quero tirar um arquivo do stage (sem perder as mudanças)
$ git reset arquivo.rb

# Quero descartar todas as mudanças em um arquivo
$ git checkout -- arquivo.rb

# Preciso trocar de branch mas não quero commitar ainda
$ git stash          # guarda as mudanças temporariamente
$ git checkout main  # vai para outro branch
$ git stash pop     # restaura as mudanças guardadas

# Desfazer o último commit (mantendo as mudanças no working dir)
$ git reset HEAD~1
Cuidado com --hard O git reset --hard descarta as mudanças permanentemente — não tem desfazer. Use com atenção e só quando tiver certeza de que não precisa do que vai ser apagado.

Aula 1

Introdução ao Ruby

Ruby é uma linguagem de programação criada em 1995 pelo japonês Yukihiro Matsumoto — conhecido carinhosamente como Matz. O objetivo dele era simples e ao mesmo tempo ambicioso: criar uma linguagem que fosse divertida de usar e que fizesse os programadores felizes. Trinta anos depois, o Ruby está na versão 4.0.5 e tem uma comunidade ativa em todos os continentes.

O framework que você vai dominar nesta mentoria — o Ruby on Rails — foi construído em cima do Ruby por David Heinemeier Hansson (DHH) e lançado em 2004. Rails e Ruby são inseparáveis: entender Ruby bem é o que vai fazer o Rails fazer sentido para você.

Por que Ruby?

Existem dezenas de linguagens de programação no mundo. O que torna o Ruby especial — especialmente para quem está começando agora?

  • Sintaxe natural. Ruby foi projetado para se ler como linguagem falada. Você vai conseguir entender o que um trecho de código faz mesmo sem ter aprendido aquele recurso ainda. Isso acelera muito o aprendizado.
  • Expressivo e produtivo. Você faz mais com menos código. A mesma lógica que levaria 20 linhas em outras linguagens pode ser expressa em 5 linhas de Ruby — sem perder clareza.
  • Orientado a objetos. Tudo em Ruby é um objeto. Até um número simples como 42 tem métodos. Você vai entender o que isso significa ao longo do curso — e vai fazer todo sentido quando chegar no Rails.
  • Tipagem dinâmica. Você não precisa declarar que uma variável é número ou texto. O Ruby descobre sozinho a partir do valor que você atribui.
  • Interpretado. Você escreve o código e executa na hora. Não precisa "compilar" nada antes de ver o resultado — o ciclo de aprendizado é muito mais rápido.
  • Ecossistema rico. Mais de 200 mil bibliotecas prontas (chamadas de gems) cobrem desde integração com APIs até geração de PDFs, processamento de imagens e muito mais. Você raramente vai precisar construir algo do zero.
A filosofia do Ruby O lema da comunidade Ruby é MINASWAN — "Matz is nice and so we are nice". É uma comunidade conhecida por acolher quem está começando, compartilhar conhecimento e ajudar sem julgamento. Você está entrando num lugar bom.

O que as pessoas dizem sobre o Ruby

David Heinemeier Hansson, criador do Ruby on Rails, descreve assim:

"Ruby é simplesmente a linguagem de programação mais bonita que já vi. Sua sintaxe permite expressar diretamente o que você quer — com código mínimo repetitivo e alta legibilidade."

E o próprio Matz completa:

Aula 1

Seu primeiro programa

Todo programador tem uma tradição: o primeiro programa que escreve exibe "Hello, World!" na tela. Vamos fazer isso — mas entendendo cada peça do que está acontecendo.

Criando o arquivo

Arquivos Ruby têm a extensão .rb. Abra o VS Code, crie uma nova pasta chamada ruby_basico e dentro dela crie um arquivo chamado hello.rb.

Escrevendo o código

Dentro do arquivo, escreva a linha abaixo e salve:

hello.rb 
puts "Olá, mundo!"

puts é a instrução que manda o Ruby exibir algo na tela. Você vai usar ela o tempo todo. O texto entre aspas é o que você quer exibir.

Executando o programa

Abra o terminal integrado do VS Code (menu Terminal → Novo Terminal), navegue até a pasta e execute:

Terminal 
$ ruby hello.rb
Olá, mundo!

Funcionou! O Ruby leu o arquivo, interpretou a instrução puts e exibiu o texto na tela.

puts vs p — qual a diferença?

Existem duas formas de exibir coisas no terminal em Ruby. É importante conhecer as duas:

exemplo.rb 
puts "Olá, mundo!"   # exibe o texto e pula uma linha
p    "Olá, mundo!"   # exibe o texto com as aspas — útil para depurar

# Resultado:
Olá, mundo!
"Olá, mundo!"
Quando usar cada um Use puts quando quiser exibir algo para o usuário. Use p quando estiver desenvolvendo e quiser ver exatamente o que está dentro de uma variável — ele mostra o tipo do dado também.

Comentários

Viu o # nos exemplos acima? Tudo depois de # numa linha é um comentário — o Ruby ignora completamente. Comentários são anotações que você deixa no código para você mesma ou para outras pessoas que vão ler depois.

comentarios.rb 
# Este programa cumprimenta a usuária
puts "Bem-vinda ao Ruby!"  # isso também é um comentário

# puts "esta linha nunca vai executar"
Aula 1

IRB — o seu playground

Criar um arquivo, escrever o código, salvar e executar pode ser um processo lento quando você está aprendendo e quer testar uma ideia rápida. Para isso existe o IRB — Interactive Ruby.

O IRB é um ambiente onde você digita código Ruby diretamente no terminal e vê o resultado na hora, sem criar arquivo nenhum. Quando você instala o Ruby, o IRB já vem junto.

Abrindo o IRB

Terminal 
$ irb
irb(main):001> # cursor esperando você digitar...

A partir daqui, tudo que você digitar será executado imediatamente ao apertar Enter:

IRB 
irb> puts "olá do IRB"
olá do IRB

irb> 10 + 5
=> 15

irb> "Ruby".upcase
=> "RUBY"
IRB vs arquivo .rb O IRB é perfeito para testar ideias e explorar a linguagem. Mas seus programas de verdade vão sempre em arquivos .rb — que você salva, versiona no Git e compartilha. Use o IRB para experimentar; use arquivos para construir.

Para sair do IRB, digite exit e pressione Enter.

Aula 2

Tipos de dados

Quando você escreve um programa, está trabalhando com informações — números, textos, listas, verdadeiro ou falso. Em Ruby, cada tipo de informação tem uma classificação chamada tipo de dado.

A boa notícia: o Ruby descobre o tipo automaticamente. Você não precisa declarar "isso é um número" — basta escrever o número. Esse comportamento chama-se tipagem dinâmica.

Os principais tipos

TipoNomeO que representaExemplos
Integer Inteiro Números sem casas decimais, positivos ou negativos 1, 42, -7, 1000
Float Decimal Números com casas decimais 3.14, 9.99, -0.5
String Texto Qualquer sequência de caracteres entre aspas "Ana", "olá!", "123"
Boolean Lógico Verdadeiro ou falso — para tomar decisões true, false
Array Lista Uma coleção ordenada de valores [1, 2, 3], ["a", "b"]
Hash Dicionário Pares de chave → valor { nome: "Ana", idade: 30 }
Symbol Símbolo Como uma string, mas imutável e mais eficiente na memória :nome, :ativo
nil Nulo A ausência de valor — "não tem nada aqui" nil

Vendo os tipos na prática

Em Ruby, todo valor tem um método chamado .class que revela qual é o seu tipo. Abra o IRB e experimente:

IRB — descobrindo tipos 
irb> 42.class
=> Integer

irb> 3.14.class
=> Float

irb> "texto".class
=> String

irb> true.class
=> TrueClass

irb> [1, 2, 3].class
=> Array

irb> { nome: "Ana" }.class
=> Hash

irb> :ativo.class
=> Symbol

irb> nil.class
=> NilClass

Tipagem dinâmica — o Ruby é inteligente

Uma das coisas mais legais do Ruby é que uma variável pode mudar de tipo ao longo do programa. Você não "declara" o tipo — o Ruby observa o valor que você atribuiu e decide:

IRB — tipagem dinâmica 
irb> dado = 10
irb> dado.class
=> Integer

irb> dado = "agora sou texto"
irb> dado.class
=> String

irb> dado = true
irb> dado.class
=> TrueClass
Atenção Apesar do Ruby ser flexível com tipos, ele não permite misturar tipos incompatíveis em operações. Tentar somar um número com um texto vai gerar um erro. Mais à frente você vai aprender como converter tipos quando precisar.

Variáveis

Uma variável é um nome que você dá a um valor para poder reutilizá-lo. Em Ruby, você cria uma variável simplesmente atribuindo um valor a ela com o sinal =:

variaveis.rb 
# Criando variáveis
nome  = "Ana"
idade = 28
ativa = true

# Usando variáveis
puts nome   # Ana
puts idade  # 28
puts ativa  # true
Convenção de nomes Em Ruby, o padrão é usar snake_case para variáveis — palavras em minúsculas separadas por underscore: nome_completo, data_nascimento, total_pedidos. Você vai ver esse padrão em todo código Ruby e Rails.

Arrays — listas de valores

Um array é uma lista ordenada. Cada elemento tem uma posição (chamada de índice), e a contagem começa sempre do zero:

arrays.rb 
frutas = ["maçã", "banana", "laranja"]

puts frutas[0]   # maçã     ← posição 0
puts frutas[1]   # banana   ← posição 1
puts frutas[2]   # laranja  ← posição 2
puts frutas[3]   # nil      ← não existe, retorna vazio

puts frutas.length  # 3 — quantos elementos tem

Hashes — dicionários de chave e valor

Um hash armazena dados em pares de chave → valor. Em vez de acessar pelo número da posição, você acessa pelo nome (a chave). Isso é muito usado no Rails:

hashes.rb 
pessoa = {
  nome:    "Ana",
  cidade:  "Porto Velho",
  ativa:   true
}

puts pessoa[:nome]     # Ana
puts pessoa[:cidade]   # Porto Velho
puts pessoa[:ativa]    # true
Símbolos como chaves de hash No exemplo acima, nome:, cidade: e ativa: são símbolos — você vai vê-los o tempo todo no Rails. Eles parecem texto, mas são imutáveis e mais eficientes para identificar chaves em hashes.
Aula 2

Operadores matemáticos

Ruby tem todos os operadores matemáticos que você conhece — e mais um par que talvez não esteja acostumada.

operadores.rb 
# Adição
puts 10 + 5    # 15

# Subtração
puts 10 - 3    # 7

# Multiplicação
puts 4 * 5     # 20

# Divisão
puts 20 / 4    # 5

# Exponenciação (potência)
puts 2 ** 8    # 256

# Módulo — o RESTO da divisão
puts 10 % 3    # 1  (10 ÷ 3 = 3, sobra 1)
puts 10 % 2    # 0  (10 ÷ 2 = 5, sobra 0 — é par!)
Módulo — para que serve na prática? O operador % retorna o resto da divisão. Parece estranho agora, mas você vai usá-lo muito: para saber se um número é par (n % 2 == 0), para distribuir itens em grupos, para criar padrões que se repetem. É uma ferramenta muito útil.

Divisão inteira — cuidado!

Quando você divide dois números inteiros em Ruby, o resultado também é inteiro — a parte decimal é descartada. Se quiser o resultado com decimais, pelo menos um dos números precisa ser float:

divisao.rb 
puts 7 / 2      # 3   ← descarta o .5 !
puts 7.0 / 2    # 3.5 ← correto, um deles é float
puts 7 / 2.0    # 3.5 ← também funciona

Operações com variáveis

calc.rb 
preco    = 150.0
desconto = 20.0
total    = preco - desconto

puts "Preço original: R$ #{preco}"
puts "Desconto: R$ #{desconto}"
puts "Total: R$ #{total}"

# Saída:
# Preço original: R$ 150.0
# Desconto: R$ 20.0
# Total: R$ 130.0
Interpolação de string Viu o #{variavel} dentro do texto acima? Isso se chama interpolação — é como você insere o valor de uma variável dentro de uma string. Muito mais elegante do que ficar concatenando com +. Use sempre aspas duplas "..." para interpolação funcionar.
Aula 2

Entrada e saída

Até agora os seus programas fazem cálculos com valores que você escreveu diretamente no código. Mas programas reais precisam receber informações do usuário — e devolver respostas. Isso se chama entrada e saída.

  • Saída — mostrar algo na tela: puts, print, p
  • Entrada — receber algo digitado pelo usuário: gets.chomp

puts vs print

Ambos exibem texto, mas com uma diferença:

saida.rb 
puts "Linha 1"   # pula linha no final
puts "Linha 2"

print "Sem pular"  # NÃO pula linha
print " — continua aqui"

# Saída:
Linha 1
Linha 2
Sem pular — continua aqui

Lendo o que o usuário digita

O gets pausa o programa e espera o usuário digitar algo e pressionar Enter. O .chomp remove a quebra de linha que o Enter insere no final:

saudacao.rb 
print "Digite seu nome: "
nome = gets.chomp

puts "Olá, #{nome}! Bem-vinda ao Ruby."

Execute com ruby saudacao.rb. O programa vai pausar esperando você digitar. Após digitar e apertar Enter, ele exibe a saudação com o nome que você digitou.

Convertendo tipos na entrada

Tudo que vem do gets é uma String — mesmo que a pessoa tenha digitado um número. Você precisa converter quando quiser fazer cálculos:

calculadora.rb 
print "Digite o primeiro número: "
num1 = gets.chomp.to_i   # .to_i converte string → inteiro

print "Digite o segundo número: "
num2 = gets.chomp.to_i

soma = num1 + num2

puts "A soma é: #{soma}"

# Exemplo de execução:
Digite o primeiro número: 15
Digite o segundo número: 7
A soma é: 22
Métodos de conversão .to_i converte para inteiro (Integer), .to_f converte para decimal (Float) e .to_s converte qualquer coisa para texto (String). Você vai usar esses três com frequência.

🕹️ Fluxo Git desta Quest — branch pela primeira vez
O que é um branch? Um branch é uma linha paralela de desenvolvimento. Você cria uma cópia do projeto, trabalha nela sem afetar a versão principal (main) e depois junta tudo de volta. É assim que times profissionais organizam o trabalho.
Terminal

# Escreva seu código, depois salve e envie

$ git commit -m "quest02: calculadora e tipos de dados"
$ git push origin quest02
